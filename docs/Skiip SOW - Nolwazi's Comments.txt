Skiip — Scope of Work (SOW)
Web Platform for Festival Ordering & Queue Skipping
Project Name: Skiip Ordering Platform
Platform Type: Web application (mobile-first)
Project Owner: Skiip Technologies

1. Project Overview
Skiip enables festival attendees to order food and drinks online from any phone, pay, and receive order status updates for pickup. The system includes:
* Customer ordering interface
* Vendor management dashboard
* Organiser / admin dashboard
* Integration with WhatsApp notifications
The current prototype (https://dk-digital-designs.github.io/skiip/index.html) is a static landing page. We require this to be transformed into a full application[NM1].

2. User Roles
RoleCapabilitiesAttendeeBrowse vendors & menus, place orders, view order status, receive notificationsVendorManage menu[NM2], view incoming orders, update order status, see reporting[NM3]Organiser/AdminManage events, vendors, users, view analytics & vendor performance
3. Functional Requirements
3.1 Attendee Ordering Interface
Goal: Mobile-first ordering flow.
Features:
* Responsive web UI (mobile/tablet/desktop)
* Vendor list page showing all active vendors
* Menu pages per vendor [NM4]with item selection, modifiers (e.g., extras), and quantity
* Order cart & summary
* Secure payment integration (Apple Pay, Google Pay, cards via Stripe/PayFast)
* Order tracking screen showing status progression
* WhatsApp and/or SMS order notifications
* Offline resilience for poor network (graceful retry, local state saving) 
* Order history (optional but recommended if login is added[NM5])
Acceptance Criteria:
* Users can place orders without timeouts
* Payments complete before final order confirmation[NM6]
* WhatsApp notifications sent automatically at key status changes
* UI matches provided designs

3.2 Vendor Portal
Goal: Enable vendors to receive, manage, and update orders.
Features:
* Vendor login/authentication
* Dashboard listing incoming orders (live)
* Order status updates (e.g., Received ? Preparing ? Ready ? Completed[NM7])
* Ability to manually update order status
* Real-time updates reflected in attendee’s app
* View order history for vendor
* Quick filtering by status (pending, preparing, ready)
* Export orders (CSV/XLS)
Acceptance Criteria:
* No manual refresh required (real-time or auto refresh)
* Status updates are pushed to attendees via API/webhooks
* Role-based access control (only vendor users see their own orders)

3.3 Organiser / Admin Dashboard
Goal: Oversee the entire event and vendor performance.
Features:
* Admin login & role management
* Event setup: Create events, set event dates, basic branding
* Vendor management: Approve/reject vendors, assign menus
* View all orders across vendors
* Analytics: Order volume, peak times, total revenue, vendor sales
* Notification logs
* Configurable commission settings (e.g., percentage per transaction)
Acceptance Criteria:
* Admin can filter reports by vendor and time
* Secure admin access with passwords and role levels

3.4 Menu & Vendor Management
Goal: Allow easy setup of vendor menus that drive the ordering interface.
Features:
* CRUD for vendor profiles (logo, description, pickup location)
* CRUD for menu items:
o Names, descriptions, prices
o Add images
o Item modifiers (e.g., size, extras)
o Inventory toggle (sold out)
* Menu item visibility toggles (e.g., hide item when sold out)
Acceptance Criteria:
* Admins & vendor users can update menus easily
* Inventory/status reflected immediately on attendee ordering screens

4. Authentication & User Management
* Decide on guest ordering vs registered account orders[NM8]
o Guest ordering can use phone number + WhatsApp for status
* Password reset & account management features if accounts exist
* Admin/Vendor users must authenticate securely[NM9]
* User sessions must be protected via HTTPS



5. Third-Party Integrations
5.1 Payments
* Stripe (or local alternative like PayFast / PayGate)
* Must be PCI-compliant
5.2 WhatsApp Messaging
* WhatsApp Business API / Twilio / Meta integration
* Send messages like:
o Order confirmation
o Payment success
o Order ready
Should support retry logic if delivery fails
5.3 Real-Time Updates
* WebSockets / Supabase Realtime / Firebase Realtime

6. Data & Storage
* Use a relational database 
* Data models include:
o Users (attendee, vendor, admin)
o Events
o Vendors
o Menu Items
o Orders & statuses
* Order history retention policies

7. Non-Functional Requirements
7.1 Performance & Scalability
* Must support peak concurrent users (10k+ per event)
* Fast API responses (<2s) 
* CDN for static assets
* Database connection pooling

7.2 Security
* HTTPS everywhere
* SQL injection & XSS prevention
* Authentication security
* Sensitive info encrypted
* Compliance with relevant data protection laws[NM10]
7.3 Responsive Design
* Must work on iOS/Android mobile browsers
* Cross-browser compatibility
7.4 Accessibility
* WCAG 2.1 AA-level accessible where possible
7.5 Documentation
* Code documentation
* Deployment instructions
* API documentation

To avoid scope creep:
* Content writing
* Photography/video production
* On-site hardware setup

11. Acceptance Criteria
The project is complete when:
* All functionality works end-to-end
* No critical bugs remain
* Performance meets agreed SLAs
* Documentation is delivered
* Code is handed over to Skiip Technologies
